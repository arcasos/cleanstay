## 1. 제품 정의

**한 줄**: STR 체크아웃(턴오버) 청소를 자동 발주·배차하는 API 레이어.

**왜 지금 이 자리인가**: 국내 청소 플랫폼(청소연구소·미소·클린벨 등) **누구도 공개 API를 제공하지 않는다**(2026-07 확인). 숙박 운영자가 "체크아웃 → 청소 자동 발주"를 하고 싶어도 붙일 API가 시장에 없다. 그 API가 되는 것이 클린콜의 자리다.

**핵심 통찰**: 실시간 소비자 청소(입주·정기)가 아니라 **STR 턴오버 청소**로 좁힌다. 턴오버는 (1) 같은 매물 반복 → 규격 표준화, (2) 체크아웃~다음 체크인이라는 명확한 시간 윈도우, (3) "다음 게스트 맞을 준비 완료" 단일 완료 기준 → 배차·단가·품질을 딱 떨어지게 잡을 수 있다.

## 2. 핵심 원칙 (절대 준수)

### 원칙 A — API-first (인프라 제품)
발주의 주력 경로는 사람이 클릭하는 UI가 아니라 **숙박 플랫폼의 이벤트가 API로 자동 생성**하는 것. 관리 콘솔·공급자 앱은 보조. 설계·문서·우선순위 모두 API + webhook 우선.

### 원칙 B — 프론트도어 (Front-door)
ARCASOS도 **내부 백도어로 붙이지 않는다.** 미래 외부 고객(야놀자·독립 STR 운영자)이 쓸 그 공개 API와 **동일한 문**으로 들어온다. 발주만 API로 넘기고, **과금은 클린콜↔호스트 직거래**(§9). 겉 연결 UX는 테넌트별로 달라도(§9 3-tier) **빌링키 볼팅 코어는 단일** — 카드/빌링키는 클린콜·PG에만, 테넌트는 토큰만.

### 원칙 C — 양면 부트스트랩 (Bootstrap)
콜드스타트 회피를 위해 **양쪽 첫 노드를 창업자가 직접 쥔다**: 수요 #1 = ARCASOS, 공급 #1 = 창업자 직영 청소업. MVP에 외부 참여자 0명으로 루프 전체를 실제로 돌린다.
- ⚠️ **내 청소업 = 부트스트랩 시드이지 최종 공급모델이 아님.** 플랫폼 가치는 결국 다수 업체 집계.
- ⚠️ **배차 공정성은 미래 신뢰 문제**(아마존 vs 셀러 구도). 배차 로직은 처음부터 "나중에 공정해질 수 있게" 설계 → v4에서 **랭킹 함수의 공정성 가중치**로 구체화(§6). 직영팀도 동일 규칙으로 콜 수신.

### 원칙 D — 지휘·감독 관계 금지 (v4 신설)
플랫폼은 **오퍼를 보내고 공급자가 수락**한다. **강제 배정(auto-assign)·업무 명령 금지.** 이를 어기면 §9의 독립계약자 구조(3.3% 원천징수·직접고용 회피)가 무너진다. 배차 UX·API·문구 전부 "콜 수신 → 수락" 형태를 유지할 것.

## 3. 정체성 — 예약형 콜 브로드캐스트 ("청소업의 배민")

| 배민 | 클린콜 |
|---|---|
| 음식점(주문 발생) | 숙박 플랫폼(체크아웃 → 청소 발주) |
| 배민 라이더 앱 | 청소업체/이모님 파트너 앱 |
| 콜 뜸 → 라이더 수락 | 청소콜 뜸 → 업체 수락 |
| 배달 완료 사진 | 청소 완료 사진 인증 |
| 정산 | 호스트 직결제 + 공급자 지급(3.3% 원천징수) |

**배민과의 결정적 차이 = 유리한 점**: 청소는 **체크아웃이 며칠 전 확정** → **예약형 콜**. 리드타임 윈도우 안에서 여러 번 재배차 여유. 초단위 매칭 엔진 불필요.

**배민이 푸는 어려운 문제 = 클린콜도 풀어야 함**: (1) 미배차 → 에스컬레이션·백업 배차, (2) 콜 분배 로직(→ v4에서 오퍼 웨이브로 확정, §6), (3) 노쇼·펑크(다음 체크인 펑크 = 치명적) → 백업 배차+마감 알림, (4) 품질/분쟁 → 완료사진+체크리스트+호스트 확인 + **공급자 정지 상태머신**(§14).

> **용어**: 공급자에게 노출되는 단어는 계속 **"콜"**. 밑단 로직이 랭크드 오퍼라는 건 내부 구현어이며 UI 문구로 쓰지 않는다.

## 4. 객체 모델 + 발주 생명주기

### 객체
| 객체 | 정의 |
|---|---|
| **테넌트(Tenant)** | 발주하는 숙박 플랫폼. API 키 발급 단위. (ARCASOS #1) |
| **호스트(Host)** | 청소값을 직접 결제하는 매물 소유자. 빌링키 등록 단위. tenant-scoped 자동 프로비저닝(§9). |
| **공급자(Provider)** | 청소를 수행하는 주체. **`type` = `business`(업체) \| `individual`(개인·이모님)** 로 분기(§14 A). 파트너 앱 사용. |
| **활동지역(ProviderServiceArea)** | 공급자의 콜 수신 범위. `{region_code, level}` 행 집합, 법정동코드 접두(§14 B). |
| **공급자 이벤트(ProviderEvent)** | append-only. 온보딩·경고·정지·해제 이력 공용 원장(§14 A/D). |
| **발주(Order)** | 하나의 턴오버 청소 건. 아래 원장 필드 포함. |
| **클레임(Claim)** | 발주에 연관된 보상/분쟁 건(§8). 골격만 지금, 정책은 나중. |

### 발주 원장 필드 (차후 보상 성립을 위해 지금 심을 것 — §8)
- **상호 참조 ID**: 클린콜 `order_id` ↔ 테넌트 `tenant_ref`(아르카 `booking_id`). **양쪽 저장.**
- **append-only 이벤트 이력**: 상태 전이·시각·행위자. 덮어쓰기 금지, 이벤트 누적.
- **금액 스냅샷**: 청구액·긴급 프리미엄·원천징수액.
- **귀책(fault)**: 실패·취소 귀책 주체(tenant/provider/host/guest). 보상 판단 근거.
- **완료 증빙**: 사진·체크리스트·타임스탬프. 클린콜이 원본, 테넌트는 참조.
- **오퍼 이력(v4 추가)**: 어느 공급자에게 몇 차 웨이브로 언제 오퍼했는지. "몰아주기 불가" 감사 로그(§6).

### 발주 생명주기
```
created            (체크인 시점에 발주 생성, trigger_type·priority 부여)
  → billing_verified (빌링키 유효성 검증 — 청구 아님, 배차 게이트 §9)
    → broadcasting   (가용 공급자 풀에 오퍼 웨이브 개시 §6)
      → accepted     (공급자 수락 → 배차 확정)
        → in_progress (체크아웃 시점 청소 시작)
          → completed (완료사진 + 체크리스트 제출)
            → confirmed (호스트 확인 또는 N시간 무이의 자동확정)
              → charged  (완료 후 빌링키 재결제 = 실제 청구)
                → paid_out (공급자에게 3.3% 원천징수 후 지급)

[예외 — 청구 없음]
  billing_verified 실패 → 배차 보류 + 호스트 "결제수단 확인" 알림
  broadcasting → (전 웨이브 미수락) → escalated → backup_dispatch
  accepted → (노쇼/취소) → reassigning → backup_dispatch
  중도퇴실(§6) → 기존 order cancelled(reason: early_checkout) + 긴급 신규 order
  * 취소·중도퇴실 건은 청구 자체가 없으므로 환불 로직 불필요
  * 각 상태 변화는 테넌트에게 webhook 통지
```

> 상태별 UI 색상 매핑은 §10 및 「디자인 시스템 전환 애드덤」 §C 표를 따른다. 상태 표기가 색으로 읽히는 것이 콘솔 UX의 핵심이다.

**Order 필드(핵심)**
- `trigger_type`: `scheduled`(정상) | `early_checkout`(중도퇴실)
- `priority`: `normal` | `urgent`
- `tenant_ref`: 테넌트측 예약 식별자(아르카 booking_id) — 상호 참조
- 시간 윈도우: `checkout_at`(청소 시작 하한) ~ `checkin_at`(마감 상한)

**Webhook (플랫폼 → 테넌트)** — 미러링·보상 대사용, 귀책·금액 스냅샷 포함
| 이벤트 | 시점 |
|---|---|
| `order.billing_verified` | 빌링키 유효성 확인(배차 게이트 통과) |
| `order.dispatched` | 공급자 수락·배차 확정 |
| `order.in_progress` | 청소 시작 |
| `order.completed` | 완료사진 제출 |
| `order.charged` | 완료 후 실제 청구 성공 |
| `order.cancelled` | 취소 (reason 포함) |
| `order.urgent_created` | 중도퇴실 등 긴급 신규 order |
| `order.failed` | 미배차/노쇼/결제실패 등 (귀책 포함) |
| `claim.updated` | 보상/분쟁 상태 변화 |

**공급측(공급자 앱)**: 콜 수신 → 수락/거절 → 시작 → 완료(사진+체크리스트). 청소업체가 API를 안 주는 현실상 직접 구축(미소 파트너·클린벨 파트너 앱 대응물).

- 인증: 테넌트 = API 키, 공급자 = 세션 토큰. service_role EF는 **caller 신원 검증 필수**(SEC-04 패턴).
- CORS: `*` 금지, allowedOrigins 화이트리스트.

## 6. 배차 엔진 설계 (제품의 심장)

**확정 모델: 예약형 랭크드 오퍼 웨이브** (v3의 "순수 브로드캐스트"를 대체 — §13 #2 해소)

```
발주 생성(체크인, 빌링키 검증 통과) → 리드타임 = checkout_at ~ checkin_at
  → 가용 공급자 풀 산출
       필터: status ∈ {active, warned}
             AND 활동지역 접두 매칭(§14 B)
             AND 가용 시간대
  → Wave 1: 랭킹 상위 소수에게 우선 오퍼 (우선 수락창 T1분)
       랭킹 = 지역근접 · 평점 · 신뢰도(노쇼율↓·완료율↑) · 공정성 가중(최근 배차 적은 순)
  → Wave 2: 미수락 → 반경 확대 + 인센티브 상향
  → Wave 3: 전체 오픈 (순수 선착순)
  → Wave 4: 미배차 → 백업 배차(직영팀 보증) + 테넌트 알림
  → 배차 후 노쇼/취소 감지 → 재배차 루프(리드타임 잔여 활용)
  → 마감 임박(체크인 N시간 전) → 강제 알림·수동 개입
```

**왜 순수 브로드캐스트보다 나은가**
1. **공정성 튜닝 가능** — 원칙 C의 "나중에 공정해질 수 있게"가 랭킹 함수 하나로 해결(선착순은 UX 전체를 갈아야 함).
2. **리드타임 활용** — 예약형이라 초단위 경쟁 불필요, 웨이브 창 여유.
3. **품질 자기강화 + 스팸↓** — 잘하는 공급자 우선, 못 잡을 콜 난사 방지.
4. **감사 로그** — 오퍼 순서·사유 기록 = "몰아주기 불가 로그" 자동 확보(§4 오퍼 이력).

**법적 필수 조건 (원칙 D)** — **강제 배정 금지, "오퍼 → 공급자 수락" 유지.** 플랫폼이 일방 배정·명령하면 지휘·감독 관계로 비쳐 §9의 독립계약자·고용리스크 회피가 깨진다.

**MVP 경로**: 초기엔 Wave 1을 생략하고 "지역·평점 필터된 풀에 오픈 오퍼"로 시작 가능. 단 **풀 산출·랭킹 필드를 처음부터 심어** 두면 나중에 Wave만 얹으면 된다.

### 중도퇴실 긴급배차 (early_checkout)
```
중도퇴실 이벤트(호스트 확인 게이트 후 발동 — 완전 자동은 오배차 리스크)
  ├─ 기존 order 자동취소 (cancelled, reason: early_checkout)
  │    broadcasting(미수락) → 조용히 취소, 손실 없음
  │    accepted(수락 완료)  → 공급자 통지 + 취소보상(테넌트 귀책, 공급자 무귀책)
  └─ 긴급 신규 order (trigger_type: early_checkout, priority: urgent)
       new checkout_at = 실제 퇴실시각, 잔여시간으로 우선순위 동적 결정
```
- **긴급 프리미엄 부담 주체 정책 필요**(호스트 카드 기본, 게스트 귀책이면 위약금 연계 검토 — §13 #3).

### 안전장치
- 매칭 기준: 활동지역 접두(필수)·가용 시간대·평점·신뢰도·(미래)매물 히스토리.
- 공정성 훅: 직영팀도 동일 규칙, 오퍼 이력 전량 기록.
- 정지(suspended)·차단(banned) 공급자는 **풀 산출 단계에서 제외**(§14 D).
- ⚠️ **`pg_net` fire-and-forget 리스크**(ARCASOS 교훈): 배차·결제·보상 같은 임계 플로우는 **재시도 + reconciliation 백스톱** 필수(§8).

## 8. ARCASOS 연동 명세 + 발주 원장(SoR)

**핵심: 클린콜 = 발주 원장의 SoR(System of Record). 아르카 = webhook 미러.**

**연동 흐름**:
```
ARCASOS 이벤트 (체크인 → 발주 / 체크아웃 → 청소 시작 트리거)
  → POST /v1/orders {property, checkin_at, checkout_at, spec, trigger_type, tenant_ref=booking_id}
    → [클린콜 배차 엔진 + 발주 원장 기록]
      → webhook (billing_verified/dispatched/completed/charged/cancelled/failed + 귀책·금액)
        → ARCASOS가 자기 booking 옆에 발주 상태 요약 미러링
          → 호스트 문의·정산 대사·차후 보상에 활용
  ※ 청소값 결제는 아르카 미경유, 클린콜↔호스트 직거래(빌링키, §9)
```

**발주 원장·보상 추적 (지금 심어야 나중에 성립)**:
- **상호 참조 ID**: 클린콜 `order_id` ↔ 아르카 `booking_id`. 양쪽 저장 → 어느 쪽 문의든 단일 지목.
- **원본/미러 분리**: 원본(상태·이력·증빙·귀책·금액)은 클린콜. 아르카는 상태 요약 미러.
- **보상 집행도 SoR 기준**: 클린콜이 원장 근거로 판단.
- ⚠️ **reconciliation 필수**: webhook 유실(pg_net) 시 미러 어긋남 → 보상 대사 깨짐. 재시도 + 아르카가 주기적으로 order 상태 pull해 대조. **보상 데이터라 fire-and-forget 금지.**

**보상 유형별 필요 데이터**(정책은 나중, 필드 자리는 지금 — `claim` 테이블 골격):
| 보상 유형 | 필요 데이터 |
|---|---|
| 청소 하자 → 호스트/게스트 보상 | 완료증빙·귀책·청구액·접수시각 |
| 미배차/펑크 → 다음 예약 피해 | 실패시각·리드타임·대체조치 |
| 파손 배상 | 공급자 식별·보험 연계·사고기록 |
| 긴급배차 프리미엄 정산 | 프리미엄 금액·부담 주체 |

**claim → 공급자 정지 연계(v4)**: 귀책(fault)이 provider인 claim은 §14 D의 정지 신호에 가중된다. 정지 판단 근거는 항상 원장(SoR).

**매핑/옵트인**: 아르카 `property`·`host` ↔ 클린콜 매핑(신원 분리). 호스트가 "자동배차 사용" 선택 시 빌링키 등록돼야 발주 가능(§9), 미등록이면 발주 차단·안내.

## 9. 결제 / 세금 / 연결 구조 (확정)

### 결제 흐름
```
게스트 청소비 → 호스트에게 정산 (아르카, 종전과 동일 — 클린콜 무관)
호스트 청소값 → 호스트가 클린콜에 직접 결제 (빌링키)
클린콜 → 공급자 지급 (개인=3.3% 원천징수 / 업체=세금계산서 정산)
```

### 결제 타이밍 — 발주 시 검증 / 완료 후 청구
```
발주(체크인)   → 빌링키 유효성 검증 (₩0 auth 또는 등록상태 확인, 청구 X)
                  유효 → 배차 진행 / 무효·미등록 → 배차 보류 + 호스트 알림
청소 시작(체크아웃)
완료 보고     → 빌링키 재결제 (실제 청구) → 공급자 지급
취소·중도퇴실 → 청구 없음 (환불 로직 불필요)
```
- **근거**: (1) 완료 후 청구 = 전자상거래법상 "용역 제공 → 대가"로 깨끗(완료사진이 근거). (2) 취소 건은 애초 청구 없음 → 환불 코드 불필요. (3) 조기청구 분쟁으로 아르카 예약에 불똥 방지.
- **유일 리스크(완료 시 카드실패)를 발주 시 유효성 검증으로 선제 차단** → "일하고 못 받는" 실패율 급감.
- **표준 참고**: 우버·호텔의 auth(예약 시 승인) & capture(이용 후 청구). 단 국내 빌링키는 보통 즉시 재결제 방식이라 순수 2단계 pre-auth 지원 여부는 **PG(PortOne) 상담 확인**(§11).

### 결제 연결 — 3-tier (겉 UX 다양, 볼팅 코어 단일)
공통 코어: **카드/빌링키 = 클린콜·PG만 보유, 테넌트 = 참조 토큰만.** PG 결제창 방식(카드가 서버 미경유)으로 PCI 부담·유출 리스크 원천 제거.

| Tier | 방식 | 대상 | 개발부담 |
|---|---|---|---|
| **1** | **Hosted Link** (결제등록 URL/QR 전달) | 롱테일 OTA·개인 운영자·PMS | ≈ 0 |
| **2** | **OAuth 리다이렉트** ("클린콜로 연결") | 자체 앱/웹 있는 중급 테넌트(**아르카 레퍼런스**) | 중 |
| **3** | **Embed SDK** (테넌트 화면에 위젯) | 최대 편의 원하는 대형 테넌트 | 최대 |

- **호스트 편의**: 간편결제(카카오·네이버·토스) 우선 노출 → 카드 16자리 입력 없이 탭 몇 번.
- **자동 프로비저닝**: 호스트는 "클린콜 회원가입" 화면을 안 봄. tenant-scoped 계정 자동 생성.
- ⚠️ **동의는 호스트 본인이 직접**: 빌링키 등록·정기결제·개인정보 처리 동의는 결제창에서 호스트가 직접 체크(전자금융거래법·개인정보보호법 — 아르카가 대신 동의 불가). 이게 아르카 책임 최소화와도 직결.
- **다중 테넌트**: 같은 사람이 아르카·야놀자 양쪽에 붙어도 초기엔 테넌트별 별도 host 레코드·별도 빌링키(격리). 통합 아이덴티티는 나중(과설계 방지).
- **한 번 등록 → 영구 자동청구**(빌링키 재결제). 연결 해제 시 빌링키 삭제(revoke).

### 세금 / 고용
- **클린콜 → 호스트에게 세금계산서/현금영수증 발행**(공급-수급 일치). 발행 시스템은 클린콜 기능.
- 공급자 = **독립 계약자**. 개인(이모님) = 3.3% 원천징수 / 업체 = 사업자 정산·세금계산서. 클린콜 = 매칭·배차 대행(직접 고용/파견 회피).
- **원칙 D(강제 배정 금지)가 이 구조의 실무적 방패**다. 배차 UX가 "명령"으로 보이면 세무·노무 프레임이 무너진다.
- 아르카는 자금 통과 역할 아님 → **전자지급결제대행·자금보유 규제 회피**.
- 청소는 지원 서비스라 숙박업·부동산중개 라인과 무관. 제3자 서비스 면책 프레임 적용.
- ⚠️ 전자금융·근로·세무는 전문가 검토 권장.

## 14. 공급자 온보딩 · 활동지역 · 배차 정지 (v4 신설)

> 섹션 번호 안정성을 위해 §1~§13 뒤에 append. 내용상으로는 §4(객체)·§6(배차)의 확장이다.

### A. 공급자 유형 & 온보딩

| 항목 | 업체(`business`) | 개인(`individual`, 이모님) |
|---|---|---|
| 식별 | 사업자등록번호 | 본인인증(휴대폰) |
| 검증 | 국세청 진위확인 + 상태조회 | 본인인증 + 계좌 실명확인 |
| 지급 | 세금계산서/사업자 정산 | **3.3% 원천징수** |
| 필수 입력 | 상호·대표자·개업일·등록증 이미지·지급계좌·활동지역 | 성명·연락처·지급계좌·활동지역 |

**사업자등록증 검증 흐름**
```
업체 입력: 사업자등록번호 + 대표자 성명 + 개업일자 (+ 등록증 이미지)
  → 국세청 "사업자등록정보 진위확인" API (공공데이터포털)
       번호+대표자+개업일 일치 여부 → valid / invalid
  → 국세청 "사업자등록 상태조회" API
       계속사업자 / 휴업 / 폐업
  → (진위 valid AND 계속사업자) → 자동 승인 후보 → 운영자 최종 승인 or 자동 active (§13 #10)
  → (invalid / 휴폐업) → 반려 + 사유 안내
```
- ⚠️ **엔드포인트·파라미터·키 발급 절차는 구현 시점에 공공데이터포털에서 재확인**(스펙 변동 가능). 진위확인 API는 일일 호출 한도가 있으므로 **결과 캐시**.
- 등록증 이미지는 육안 대조·보관용. 진위의 근거는 어디까지나 API(자동 OCR은 범위 밖, §12).

**온보딩 상태**
```
registered (가입) → pending_review (검증 대기) → active (배차 대상)
                                              └ rejected (반려, 사유 표기)
```

**DB 델타**
- `providers`: `id`, `type(business|individual)`, `status`(온보딩 + 정지 상태머신 공유), `display_name`, `phone`, `bank_account`, `rating`, `created_at`
- 업체 전용: `business_no`, `rep_name`, `open_date`, `biz_verify_status(valid|invalid|pending)`, `biz_status(active|closed|suspended_tax)`, `license_image_url`
- `provider_service_areas`: `provider_id`, `region_code`, `level(city|gu|dong)` — B 참조
- `provider_events`(append-only): `provider_id`, `event(registered|verified|rejected|activated|warned|suspended|reinstated|banned)`, `reason`, `actor`, `at` — 온보딩·정지 이력 공용 원장

### B. 활동지역 모델 (법정동코드 접두 계층)

공급자가 **서울 전역 / 특정 구 / 특정 동**을 자유 조합으로 고른다. 저장·매칭은 **법정동코드 접두(prefix)**.

```
법정동코드 계층 (행정표준코드관리시스템 기준)
  서울특별시 = "11"          (2자리 접두)
  강남구     = "11680"       (5자리 접두)
  역삼동     = "1168010100"  (전체 코드)

공급자 선택 저장 = { region_code, level } 행들
  "서울 전역" → { "11", city }
  "강남구"    → { "11680", gu }
  "역삼동"    → { "1168010100", dong }
```

**매칭 규칙**(§6 풀 산출에서):
```
property.region_code(10자리)가 공급자 선택 접두 중 하나로 시작하면 → 커버리지 있음
  매물 = 1168010100(역삼동)
    공급자 A "11"(서울전역)  → startsWith → 매칭 O
    공급자 B "11680"(강남구) → startsWith → 매칭 O
    공급자 C "11710"(송파구) → 불일치     → 매칭 X
```
- 장점: 계층 확장/축소가 접두 길이로 자연 표현, 매칭이 문자열 접두 검사 한 줄.
- ⚠️ 실제 코드값은 행정표준코드관리시스템에서 가져와 시드 데이터로 적재(위 예시는 형식 설명용).

### C. 배차 정지 (공급자 상태머신)

**"청소 결과에 큰 불만이 누적된 공급자를 배차 풀에서 제외"** 하는 안전장치.

```
active ─(경고 사유)→ warned ─(임계 초과/중대 사유)→ suspended ─(재심)→ reinstated(→active)
   └────────────────(중대·반복)───────────────────────────────→ banned (영구)

정지(suspended) 효과:
  - 배차 풀 산출에서 제외 → 신규 콜 안 뜸
  - 진행 중(accepted/in_progress) 발주는 그대로 마무리 (중도 회수 시 재배차 비용 발생)
  - 정지 기간 / 무기한 선택 가능
```

**트리거**
- **수동**: 운영자가 claim·불만 검토 후 정지 (사유 필수 입력).
- **자동(임계)**: 평점 하락(예 4.0↓) / 노쇼 N회 / claim 발생률 / 완료증빙 미제출률 → 자동으로 `warned` 또는 `suspended` **후보**로 올려 운영자 확인. 구체 수치는 §13 #11.

**공정성·법적 요건**
- 정지·해제 모두 `provider_events`에 **append-only 로그**(사유·행위자·시각). 임의 차단이 아님을 감사 가능하게.
- 정지 사유 유형은 **약관에 근거 명시**(독립계약자 계약상 일시중지/해지 근거). 없으면 부당 배차중단 분쟁.
- **완전 자동 영구밴 금지** — 자동 판정은 사람 확인 게이트를 거친다(오탐 방지).
- claim 연계: 귀책이 provider인 claim은 정지 신호에 가중. 판단 근거는 원장(SoR, §8).

### D. 관련 화면
공급자 가입(유형 선택 / 업체 등록 / 개인 등록 / 활동지역 선택)과 콘솔 공급자 관리·정지 모달의 상세 UI 스펙 및 Readdy 프롬프트는 「디자인 시스템 전환 애드덤」 §G(G1~G5)에 있다. **§10의 토큰 규칙을 벗어나는 프롬프트는 사용 금지**(구 Linear Dark 프롬프트는 폐기됨).
